## 📌 一、必须遵守的 5 条铁律

| 铁律 | 说明 |
|------|------|
| 提交前必读文档 | 每次提交前阅读：后端开发文档.md、开题报告、application.yml |
| 注释即文档 | 关键算法、安全逻辑、数据处理必须写注释，禁止敷衍 |
| 每日代码审查 | 站会检查：注释完整性、权限配置、数据清洗逻辑 |
| 分支管理 | 使用 `feat/xxx-日期`、`fix/xxx-日期`，禁止 `branch1` 等无意义命名 |
| 文档同步 | 接口/逻辑/表结构变更后，同步更新对应文档和 Hive 脚本 |

## 💬 二、注释规范（强制执行）

### 2.1 标准模板

```java
/**
 * [功能简述]
 * 
 * @param [参数名] [参数说明]
 * @return [返回值说明]
 * @throws [异常类型] [异常说明]
 * 
 * 依据：开题报告 [章节号] - [具体需求]
 * 示例：/api/auth/login
 */
```

### 2.2 必须注释的 4 类代码

| 代码类型 | 必须包含内容 | 示例要点 |
|----------|--------------|----------|
| 核心服务 | 业务逻辑 + 开题报告依据 | 组织效能计算、Hive 查询说明 |
| 安全配置 | 权限控制依据 | HR_ADMIN/DEPT_HEAD 访问范围 |
| Hive 交互 | 表结构 + 分区逻辑 | hr_analytic_data、period 分区 |
| 数据清洗 | 清洗规则说明 | 空值→0.0、异常值→100% |

### 2.3 正确与错误示例

❌ **错误**：`// 计算数据`  
✅ **正确**：`// 计算组织效能指数（依据开题报告 1.2 节）`

❌ **错误**：无参数、无返回值说明  
✅ **正确**：

```java
/** 
 * 计算员工流失率预警阈值
 * 规则：当流失率 > threshold 时触发预警
 * @param threshold 阈值（百分比，如 8.0 表示 8%）
 * @return 流失率数据列表
 * 依据：开题报告 1.2 节"员工流失预警"
 */
```

## 🤝 三、开放开发习惯（团队协作）

### 3.1 分支命名

```bash
# 正确
git checkout -b feat/user-login-20260221
git checkout -b fix/department-filter-20260221

# 错误
git checkout -b branch1
```

### 3.2 代码修改后必须同步

- 后端开发文档.md（新增/变更接口）
- 开题报告.pdf（功能实现进度）
- src/main/resources/hive/（Hive 表结构变更）

### 3.3 提交信息格式

```
feat: 实现用户登录模块 (依据开题报告 3.2.1 节)
fix: 修复部门负责人越权查询 (依据开题报告 3.2.1 节)
```
